## Кейс (хакатон)

**Название:** Визуальный цифровой двойник локомотива с индексом здоровья и потоковой телеметрией

**Суть:** Full-stack дашборд, который в реальном времени принимает телеметрию тепловоза, рассчитывает «Индекс здоровья» (0–100) и визуализирует параметры с алертами.

**Критерии оценки:**
- UI/UX дизайн — 30%
- Работа в реальном времени (WebSocket, latency, отказоустойчивость) — 35%
- Архитектура backend — 25%
- Демо и инженерная культура — 10%

## Локомотив: ТЭ33А

- Грузовой тепловоз GE/Wabtec (Evolution Series)
- Колея 1520 мм, макс. скорость 120 км/ч
- Масса 138 тонн, мощность 4500 л.с.
- Асинхронные тяговые двигатели переменного тока
- Дизель-электрическая тяга

Данные получены из интервью с действующим машинистом.

## Три роли / вида

### 1. Машинист — "Кабина" (`/cabin`)
- Один конкретный локомотив
- Крупные показания (смотрит за рулём)
- Индекс здоровья — главный виджет
- Скорость крупно, алерты яркие
- Read-only — ничего не настраивает

### 2. Диспетчер — "Центр управления" (`/dispatch`)
- Список всех локомотивов с их статусом
- Клик → подробности конкретного локомотива
- История алертов, replay 5–15 мин
- Экспорт отчётов PDF/CSV
- Карта с положением всех локомотивов

### 3. Администратор — "Настройки" (`/admin`)
- Настройка порогов (при какой температуре алерт)
- Настройка весов индекса здоровья
- Управление пользователями
- Логи, метрики, health-check
- Swagger/API документация

Авторизация: роли `driver`, `dispatcher`, `admin`.

## Индекс здоровья

**Формула:**
```
HI = Σ (wᵢ × scoreᵢ)
```
- `scoreᵢ = f(value, norm_min, norm_max, warning, critical)` — нормализация 0–100
- `wᵢ` — веса, выше для параметров, ведущих к снятию нагрузки
- Штрафы за активные алерты (заземление, боксование) — бинарные, сразу роняют индекс

**Категории:**
- **Норма (A / зелёный):** HI ≥ 80
- **Внимание (C / жёлтый):** 50 ≤ HI < 80
- **Критично (E / красный):** HI < 50

**Объяснимость:** при клике на индекс — top-5 факторов с вкладом каждого.

**Пороги из реальных данных:**
| Фактор | Норма | Внимание | Критично |
|---|---|---|---|
| Темп воды двигателя | 71–92 °C | <71 или >92 | >115 °C |
| Темп масла | — | приближение к 96 | >96 °C |
| Расход воздуха | 1500–3200 | у границ | вне диапазона |
| Заземление цепей | норма | — | пробой изоляции |

Конфигурация порогов и весов — в БД/файле, без перекомпиляции.

## Расчёт нагрузки

**Реальный масштаб:** 1700 локомотивов, 50 000 составов.

```
Нормальный режим: 1 700 RPS, ~1.4 МБ/сек
Пиковый (x10):   17 000 RPS, ~14 МБ/сек
Хранилище 72ч:   ~350 ГБ сырых → ~30–50 ГБ с compression
WebSocket fan-out: ~1 820 соединений, ~5 100 msg/сек исходящих
```

**End-to-end latency budget:** ~48 мс (бюджет ТЗ: 500 мс, запас x10).

## Архитектура

Двухуровневая: **Edge (борт) + Cloud (центральный бэкенд)**.

```
БОРТ ЛОКОМОТИВА (автономно, без сети):
  Датчики → Edge Processor (бортовой ПК) → Дашборд машиниста (localhost)
                 ↓ sync (при наличии сети)

ЦЕНТРАЛЬНЫЙ БЭКЕНД (K3s кластер):
  Edge sync (1700) → NLB/Nginx → Ingestion Service (stateless, HPA)
    → Kafka (Strimzi, 3 брокера, 12 партиций, 72ч retention)
      → Processor: Normalizer + Health Index Calculator + Alert Detector
        → PostgreSQL + TimescaleDB (primary + replica, 72ч retention)
        → Redis (current state cache + pub/sub)
          → WS Hub (stateless, HPA) → WebSocket → диспетчер/админ
          → REST API (stateless, HPA) → HTTP → диспетчер/админ
```

**Edge Processor** на борту — считает HI, алерты, LCAS голос локально.
Машинист **всегда** видит данные, даже без сети. При восстановлении — sync.

Все application-сервисы **stateless**, масштабируются через K8s HPA.
Stateful-сервисы (Kafka, PostgreSQL, Redis) управляются K8s-операторами.

- **Strimzi** — Kafka (3 брокера)
- **CloudNativePG** — PostgreSQL + TimescaleDB (primary + replica)
- **Redis** — StatefulSet

- **cert-manager** — Let's Encrypt TLS
- **Prometheus + Grafana** — мониторинг

### HPA авто-скейлинг
| Сервис | min pods | max pods | scale trigger |
|---|---|---|---|
| ingestion | 2 | 10 | CPU > 70% |
| processor | 2 | 10 | CPU > 70% |
| ws-hub | 2 | 8 | CPU > 60% |
| rest-api | 2 | 6 | CPU > 70% |
| frontend | 2 | 2 | фиксированно |

### Портируемость
Всё — open-source, без vendor lock-in. Docker-контейнеры + K8s манифесты работают на любом железе: VPS, bare metal, on-premise серверы заказчика. Миграция = смена `.env` файла.

## LCAS — Locomotive Crew Alerting System

Система алертов авиационного типа (аналог EICAS/ECAM):

- **Три уровня:** WARNING (красный, голос, повтор) / CAUTION (жёлтый, голос) / ADVISORY (тихий тон)
- **Приоритетная очередь:** WARNING прерывает CAUTION, подавление мелких алертов
- **Голосовой синтез:** Web Speech API, русский язык, "ТЕМПЕРАТУРА ВОДЫ КРИТИЧНА. СНЯТЬ НАГРУЗКУ"
- **Master Warning / Master Caution:** кнопки как в кокпите, acknowledge останавливает повтор
- **Предиктивный HI:** линейная экстраполяция, "HI < 50 через ~12 мин"

Аналогия: мы строим **EICAS для локомотивов** — авиационный стандарт безопасности, впервые в ЖД.

## Simulator Control Panel

Отдельный инструмент (`/simulator`) для управления виртуальным тепловозом:

- **Слайдеры** для каждого параметра с цветовыми зонами
- **Карта с маршрутом** — локомотив едет по маршруту с текущей скоростью
- **3 режима:** ручной / авто (шум ±2%) / готовый сценарий
- **7 сценариев:** перегрев воды, перегрев масла, утечка воздуха, разгон-торможение, тяжёлый участок, каскадный отказ, нормальная работа
- **Highload-тест:** генерация 1–1700 виртуальных локомотивов
- Данные идут через тот же pipeline — бэкенд не различает источник

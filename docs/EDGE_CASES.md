## Данные и датчики

### 1. Датчик заклинил (stuck sensor)

Датчик шлёт одно и то же значение. В реальности шум ±0.5°C есть всегда.

```
12:00  water_temp = 85.0
12:01  water_temp = 85.0
12:02  water_temp = 85.0
12:03  water_temp = 85.0   ← 4 минуты одно число — датчик мёртв
```

**Опасность:** Ложное спокойствие — реальная температура может быть 120°C.

**Детекция:** Дисперсия за 2 мин = 0 → stuck.

**Реакция системы:**
- CAUTION: "ДАТЧИК ТЕМПЕРАТУРЫ ВОДЫ — НЕТ ИЗМЕНЕНИЙ"
- HI: вес этого параметра обнуляется (не учитывать сломанный)
- Визуально: значение серым с пометкой "⚠ stale"
- Data Quality Score → 0% для этого параметра

### 2. Spike / outlier от датчика

Одиночный выброс — не должен вызывать алерт.

```
12:00  water_temp = 85.0
12:01  water_temp = 85.2
12:02  water_temp = -999.0   ← мусор
12:03  water_temp = 85.1
```

**Реакция системы:**
- EMA-фильтр сглаживает разовые выбросы
- Значение вне физически возможного диапазона → отбрасывается:
  - Температура: < -50 или > 500 → drop
  - Давление: < 0 или > 2000 кПа → drop
  - Скорость: < 0 или > 200 км/ч → drop
- Data Quality Score снижается
- Если мусор 3+ раз подряд → CAUTION: "ДАТЧИК НЕИСПРАВЕН"

### 3. Пакет без поля

```json
{
  "locomotive_id": "TE33A-0142",
  "speed_kmh": 72,
  "water_temp_inlet": 85
  // oil_temp_outlet — ОТСУТСТВУЕТ
}
```

**Реакция:**
- Бэкенд не падает — отсутствующее поле = `null`
- HI считается без этого параметра (вес перераспределяется)
- Визуально: "—" вместо значения
- Если поле отсутствует 5+ пакетов подряд → ADVISORY

### 4. Дубликат пакета

Сеть нестабильна → edge-agent отправил → не получил ACK → отправил снова.

**Реакция:**
- Дедубликация по `locomotive_id + timestamp`
- Если timestamp совпадает ±100 мс → второй пакет игнорируется
- Логируется в метриках: `telemetry_duplicates_total` counter

### 5. Пакеты не по порядку

```
12:00:03  получен пакет с timestamp 12:00:01
12:00:04  получен пакет с timestamp 12:00:03
12:00:05  получен пакет с timestamp 12:00:02   ← опоздал
```

**Реакция:**
- HI считается по последнему по timestamp, не по порядку получения
- Буфер reordering: 5 сек окно
- PostgreSQL: запись по timestamp (не по insertion time)
- Графики: точки ставятся по timestamp, не по времени получения

## Сеть и синхронизация

### 6. Thundering Herd — 1700 reconnect одновременно

Сервер перезапустился → все 1700 локомотивов переподключаются одновременно.

```
00:00  0 подключений
00:01  1700 подключений одновременно
00:02  все шлют накопленный буфер → x10–x100 RPS на 30 секунд
```

**Реакция:**
- Ingestion: rate limiter на locomotive_id (max 10 RPS на один)
- Edge-agent reconnect с jitter: random(0–30 сек) задержка
- Kafka: поглощает пик, Processor догоняет постепенно
- HPA: масштабирует ingestion pods
- В Ingestion: очередь accept с backpressure

### 7. Буфер переполнен (72ч оффлайн)

Edge-agent копит данные 72ч → SQLite ~2 ГБ → сеть всё ещё нет.

**Реакция:**
- Кольцевой буфер: старые данные перезаписываются
- Алерты никогда не удаляются (отдельная таблица)
- При синхронизации: "данные за последние 72ч, алерты за весь период"
- Если буфер > 90% — ADVISORY: "БУФЕР ДАННЫХ ПОЧТИ ПОЛОН"

### 8. Сеть мерцает (flapping)

Сеть появляется на 3 сек, пропадает на 10, появляется на 5...

```
00:00  ONLINE → sync
00:03  OFFLINE → sync прерван
00:13  ONLINE → sync с checkpoint
00:18  OFFLINE → ...
```

**Реакция:**
- Sync с checkpoint: запоминаем последний успешно отправленный timestamp
- UI: не переключать "ONLINE"/"OFFLINE" при flapping
- Если >3 переключений за минуту → показывать "НЕСТАБИЛЬНОЕ СОЕДИНЕНИЕ"
- Debounce: считать "online" только после 10 сек стабильного соединения
- Reconnect с exponential backoff: 1с, 2с, 4с, 8с, 16с, max 30с

## Время

### 9. Clock drift — рассинхронизация часов

Часы на бортовом ПК отстают на 5 минут от серверного времени.

**Реакция:**
- NTP-синхронизация при каждом подключении к сети
- Бэкенд: принимает `client_timestamp`, записывает и свой `server_received_at`
- При drift > 30 сек → ADVISORY: "РАССИНХРОНИЗАЦИЯ ЧАСОВ"
- HI и алерты считаются по `server_received_at`
- Графики: offset коррекция при рендеринге

### 10. Часовые пояса

Казахстан = UTC+5 (запад) и UTC+6 (восток). Локомотив едет между зонами.

**Реакция:**
- Все timestamp внутри системы — **UTC**
- Отображение — по локальному времени пользователя (`Intl.DateTimeFormat`)
- Retention policy (72ч) — по UTC
- Расписание (#32) — в локальном времени станции

## Масштаб и нагрузка

### 11. Один лок шлёт x100 частоту

Сбой в edge-agent → TE33A-0231 шлёт 100 пакетов/сек вместо 1.

**Реакция:**
- Rate limiter на Ingestion: max 10 RPS на locomotive_id
- Превышение → 429 Too Many Requests → edge-agent снижает частоту
- ADVISORY в /admin/system: "TE33A-0231 rate limited (100→10 RPS)"
- Остальные 1699 локомотивов не затронуты
- Метрика: `telemetry_rate_limited_total{locomotive_id}`

### 12. PostgreSQL lag — запись отстаёт

Processor пишет batch в PostgreSQL, при пике batch растёт, запись отстаёт на 30+ сек.

**Реакция:**
- Realtime (Redis pub/sub → WS Hub) не затронут
- REST API для "последних 30 сек" → читает из Redis (горячие данные)
- REST API для "старше 30 сек" → читает из PostgreSQL
- Алерт в /admin/system: "DB write lag: 30 sec"
- Circuit breaker: если lag > 60 сек → Processor в режим "только Redis"
- Kafka consumer offset не коммитится → при восстановлении БД перечитает

### 13. Redis OOM — память заполнена

1700 лок × состояние + pub/sub → Redis под давлением.

**Реакция:**
- `maxmemory-policy: allkeys-lru` — старые ключи вытесняются
- Мониторинг: алерт если memory > 80%
- Ключи `locomotive:{id}:health` (HI) помечены высоким приоритетом
- Алерт в /admin/system: "Redis memory: 85%"

## Человеческий фактор

### 14. Машинист не отвечает на WARNING

LCAS кричит "СНЯТЬ НАГРУЗКУ" 60 секунд. Машинист не нажал "Принял".

**Возможные причины:** заснул, покинул кабину, не слышит, медицинская проблема.

**Реакция — эскалация:**

```
0 сек     WARNING → LCAS голос → Master Warning мигает
5 сек     повтор голоса
10 сек    повтор голоса
...
30 сек    ЭСКАЛАЦИЯ → диспетчеру: "WARNING без подтверждения >30 сек"
          Диспетчер видит: красная пометка "МАШИНИСТ НЕ ОТВЕЧАЕТ"
60 сек    CAUTION у диспетчера: "МАШИНИСТ НЕ ОТВЕЧАЕТ >60 сек"
          Telegram-уведомление (если включено)
120 сек   Автоматическое снижение нагрузки (если интегрировано)
```

Аналог в авиации: нереагирование на GPWS "PULL UP" записывается в FDR и расследуется.

На дашборде диспетчера:

```
TE33A-0231  HI: 32 [R]  ⚠ МАШИНИСТ НЕ ОТВЕЧАЕТ (47 сек)
```

### 15. Смена машиниста во время инцидента

```
12:34  WARNING: перегрев, машинист Иванов → acknowledge
12:45  Смена: Иванов → Петров (новый логин)
12:46  Петров видит acknowledged алерт, но не знает контекст
```

**Реакция:**
- Shift Handover: при смене логина → сводка текущих проблем
- Acknowledged алерты снова становятся ACTIVE для нового машиниста
- LCAS: "ВНИМАНИЕ. АКТИВНЫЙ ИНЦИДЕНТ. ТЕМПЕРАТУРА ВОДЫ ПОВЫШЕНА"
- Контекст: "Инцидент начался 12:34, предыдущий машинист подтвердил"
- Timeline инцидента (#40) показывает полную историю

### 16. Два диспетчера на одном локомотиве

Диспетчер A и B открыли TE33A-0231. Оба нажимают "Принял".

**Реакция:**
- Optimistic locking: первый acknowledge записывается
- Второй: "Уже подтверждено (dispatcher_a, 12:34:56)"
- Audit log: оба действия записаны
- WebSocket: оба клиента получают обновление статуса алерта

## Физические условия

### 17. GPS в тоннеле

Поезд в тоннеле → GPS пропал → позиция не обновляется.

**Реакция:**
- Dead reckoning: позиция экстраполируется по последней скорости и направлению

```python
def dead_reckoning(last_position, speed_kmh, bearing, seconds):
    distance_km = speed_kmh / 3600 * seconds
    new_position = destination(last_position, distance_km, bearing)
    return new_position
```

- На карте: маркер с пометкой "⚠ GPS lost, расчётная позиция"
- Data Quality Score GPS → 0%
- При выходе из тоннеля → скачок позиции, плавная интерполяция (ease)

### 18. Холодный старт (−40°C)

Зима в Казахстане: −40°C. Дизель холодный, все температуры ниже нормы.

```
Обычный режим:     вода 71–91°C → норма
Холодный старт:    вода 15°C → это НЕ warning, это прогрев
```

**Реакция:**
- Автоопределение фазы: скорость = 0 + все температуры < 40°C → режим "ПРОГРЕВ"
- В режиме прогрева: пороги warning/critical сдвигаются
  - Вода < 71°C → норма (вместо warning)
  - Масло < 72°C → норма
- HI в режиме прогрева: отдельная формула (пониженные веса температур)
- Визуально: "РЕЖИМ ПРОГРЕВА" в header
- Автовыход из режима: все температуры > norm_min → "РАБОЧИЙ РЕЖИМ"

## Обработка на уровне кода

### Валидация входящего пакета (Ingestion)

```python
def validate_packet(data: dict) -> tuple[dict, list[str]]:
    errors = []

    if "locomotive_id" not in data:
        return None, ["missing locomotive_id"]

    if "timestamp" not in data:
        data["timestamp"] = datetime.utcnow().isoformat()
        errors.append("missing timestamp, using server time")

    # физические границы (не пороги алертов, а возможные значения)
    bounds = {
        "speed_kmh": (-1, 200),
        "water_temp_inlet": (-50, 500),
        "water_temp_outlet": (-50, 500),
        "oil_temp_inlet": (-50, 300),
        "oil_temp_outlet": (-50, 300),
        "water_pressure_kpa": (-10, 2000),
        "oil_pressure_kpa": (-10, 2000),
        "fuel_level": (-1, 101),
    }

    for field, (lo, hi) in bounds.items():
        if field in data and data[field] is not None:
            if data[field] < lo or data[field] > hi:
                errors.append(f"{field}={data[field]} out of physical bounds")
                data[field] = None  # отбрасываем невозможное значение

    return data, errors
```

### Детекция stuck sensor (Processor)

```python
class StuckDetector:
    def __init__(self, window_seconds=120, min_variance=0.01):
        self.window = window_seconds
        self.min_variance = min_variance
        self.history = defaultdict(list)

    def check(self, loco_id, param, value, timestamp):
        key = f"{loco_id}:{param}"
        self.history[key].append((timestamp, value))

        # оставляем только последние window секунд
        cutoff = timestamp - self.window
        self.history[key] = [(t, v) for t, v in self.history[key] if t > cutoff]

        if len(self.history[key]) < 10:
            return False  # недостаточно данных

        values = [v for _, v in self.history[key]]
        variance = sum((v - sum(values)/len(values))**2 for v in values) / len(values)

        return variance < self.min_variance  # True = stuck
```

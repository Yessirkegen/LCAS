## Детали по каждому улучшению

### 1. LCAS — Locomotive Crew Alerting System

Полная спецификация: [docs/LCAS.md](LCAS.md)

Авиационного типа система оповещения:
- Три уровня: WARNING / CAUTION / ADVISORY
- Голосовой синтез речи (Web Speech API, русский)
- Приоритетная очередь (WARNING прерывает CAUTION)
- Master Warning / Master Caution кнопки (как в кокпите)
- WARNING повторяется каждые 5 сек до acknowledge
- Подавление мелких алертов при активном WARNING

### 2. Предиктивный Health Index

Линейная экстраполяция тренда HI за последние 5 минут:

```
Текущий HI: 72 (Внимание)
Тренд: -2.3 в минуту
Прогноз: HI < 50 через ~12 мин
[Рекомендация: снизить нагрузку]
```

На графике — пунктирная линия, уходящая в будущее.
LCAS CAUTION: "ВНИМАНИЕ. ПРОГНОЗ СНИЖЕНИЯ ИНДЕКСА ЗДОРОВЬЯ"

### 3. Simulator Control Panel

Полная спецификация: [docs/SIMULATOR.md](SIMULATOR.md)

- Слайдеры для каждого параметра с цветовыми зонами
- Карта с маршрутом и движением
- 3 режима: ручной / авто / сценарий
- 7 готовых сценариев (перегрев, утечка, каскадный отказ)
- Highload-генератор (1–1700 виртуальных локомотивов)
- Лог отправки пакетов

### 4. Звуковой дизайн алертов

Web Audio API, разные тоны по типу:
- WARNING: тройной резкий зуммер 1200 Гц (square wave)
- CAUTION: одинарный гонг 800 Гц (sine wave)
- ADVISORY: мягкий тон
- Resolved: позитивный тон
- Соединение восстановлено: мягкий аккорд

### 11. Circuit Breaker

Если PostgreSQL упал — Processor продолжает работать:
- Данные в Redis (кэш), в Kafka (ждут)
- WS Hub продолжает отдавать из кэша
- Алерт "DB unavailable" в /admin/system
- При восстановлении — Kafka consumer догоняет

Демо: "отключаю БД — дашборд продолжает работать".

### 16. Anomaly Detection

Не только пороговые алерты, а статистические — "параметр ведёт себя нетипично".

```
Обычно при скорости 80 км/ч температура масла = 74±3°C.
Сейчас при 80 км/ч температура масла = 82°C.
Это нетипично → CAUTION: "АНОМАЛИЯ. ТЕМПЕРАТУРА МАСЛА"
```

Скользящее среднее + стандартное отклонение за 24ч. Если значение > μ + 3σ при
тех же условиях (скорость, нагрузка) — аномалия.

### 29. Реальная карта ЖД-путей Казахстана

Вместо абстрактной линии A→B — реальная сеть железных дорог Казахстана.
Локомотив едет **по настоящим рельсам** с поворотами, кривыми, стрелками.

**Источники данных:**

| Источник | Что даёт | Формат |
|---|---|---|
| OpenRailwayMap | Готовые тайлы всех ЖД-путей мира | Растровый слой поверх карты |
| Overpass API (OSM) | Геометрия ЖД-путей Казахстана | GeoJSON, можно стилизовать |
| OSRM (train profile) | Маршруты вдоль реальных рельсов | GeoJSON LineString |
| Overpass API (stations) | Станции с названиями | GeoJSON Points |

**Ключевые маршруты КТЖ:**

| Маршрут | Расстояние | Значимость |
|---|---|---|
| Астана → Караганда | ~230 км | Главный грузовой коридор |
| Алматы → Шу → Астана | ~1200 км | Магистраль Север–Юг |
| Актау → Бейнеу | ~400 км | Нефтяной коридор |
| Костанай → Астана | ~700 км | Зерновой коридор |
| Достык → Алматы | ~700 км | Транзит Китай–Европа |

**Движение по маршруту:**
- GeoJSON LineString из OSRM — реальная геометрия рельсов (~2000–5000 точек на маршрут)
- Turf.js `@turf/along` — интерполяция позиции на расстоянии N км от начала
- Turf.js `@turf/bearing` — поворот маркера по направлению движения
- Скорость со слайдера → расстояние за 1 сек → новая позиция на маршруте

**Что видно на карте:**

Вид машиниста (`/cabin`):
- Серые линии: вся ЖД-сеть (OpenRailwayMap или GeoJSON)
- Яркая линия: текущий маршрут
- Маркер локомотива: повёрнут по направлению движения
- Станции на маршруте с названиями
- Зоны ограничения скорости (цветные участки)

Вид диспетчера (`/dispatch`):
- Вся ЖД-сеть Казахстана
- 1700 цветных точек (зелёный/жёлтый/красный по HI)
- Кластеризация при zoom-out (50+ лок рядом → число)
- Клик на точку → popup с HI, скоростью, алертами
- Слои: [x] ЖД-пути [x] Локомотивы [x] Станции [ ] Ограничения

**Стек:**
- MapLibre GL JS (WebGL, 1700 маркеров)
- OpenRailwayMap tiles (растровый слой ЖД)
- Turf.js (геометрические операции)
- GeoJSON файлы маршрутов (предзагружены)

**Подготовка данных (один раз):**
- Скачать маршруты через OSRM → `public/geo/route_astana_karaganda.json`
- Скачать ЖД-сеть через Overpass → `public/geo/kz_railways.json`
- Скачать станции через Overpass → `public/geo/kz_stations.json`

**Бонусы:**
- Маркер поворачивается по направлению движения (bearing)
- При highload-тесте 1700 точек разбросаны по реальной сети — выглядит как настоящая диспетчерская
- Станции отображаются при zoom-in

### 30. Профиль пути — уклоны и высоты

Визуализация рельефа маршрута. На подъёме нагрузка выше → температуры растут.

```
Высота
 500м ─          ╱╲
 400м ─    ╱━━━━╱  ╲━━━━╲
 300м ─ ━━╱                ╲━━━
          Астана    подъём    Караганда
          0 км      120 км     230 км
                      ▲ вы тут
```

Диспетчер видит: "через 10 км подъём, при текущей нагрузке температура может вырасти".
Данные высот из OpenStreetMap или SRTM (бесплатные).

### 31. Состав — вагоны + вес

Схема состава: локомотив + N вагонов с типом груза и весом.

```
[ТЭ33А-0142]═══[60т]═══[60т]═══[58т]═══[62т]═══[пуст]═══[45т]
  локомотив     зерно   зерно   зерно   зерно   порожн.  уголь
                Общий вес состава: 423 тонн
```

Вес влияет на расход топлива, тяговое усилие и температуры.
Для симулятора — слайдер "вес состава: 200–6000 тонн".

### 32. Расписание и отклонения

Плановое vs фактическое время прибытия на станции:

```
Станция       План     Факт      Отклонение
Астана        06:00    06:00     вовремя
Сороковая     06:35    06:38     +3 мин
Агадырь       07:45    07:52     +7 мин ← нарастает
Караганда     09:30    ???       прогноз: 09:41 (+11 мин)
```

Причины отклонений связываются с HI: "опоздание из-за снижения скорости после алерта".

### 33. Погода на маршруте

Реальная погода через OpenWeatherMap API (бесплатно до 1000 запросов/день):

```
Астана:     -5°C, ветер 15 м/с, снег
120 км:     -8°C, ветер 20 м/с, метель ⚠
Караганда:  -3°C, облачно
```

Влияние: холод → долгий прогрев, жара → перегрев, метель → ограничение скорости.
На карте — иконки погоды вдоль маршрута.

### 37. Graceful Degradation — приоритеты данных

При слабом соединении (2G) — система решает что слать первым:


### 42. Edge Computing — автономная работа на борту

Safety-critical: машинист не должен терять данные и алерты при потере сети.
Принцип из авиации: все приборы кокпита работают локально, связь с землёй — опционально.

**Архитектура:**

```
┌─── ЛОКОМОТИВ (автономно, без сети) ──────────────────────┐
│                                                           │
│  Датчики ──► Edge Processor ──► Локальный дашборд        │
│              (бортовой ПК)      машиниста                 │
│              │                                            │
│              ├─ HI расчёт (локально)                     │
│              ├─ Алерты + LCAS голос (локально)           │
│              ├─ Графики (локально)                       │
│              └─ Буфер данных (SQLite, до 72ч)           │
│                                                           │
│              Если сеть есть:                              │
│              └──► sync ──► Центральный бэкенд            │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

**Бортовой мини-сервер** (Raspberry Pi / встроенный ПК):

```
edge-processor (Python/Go, ~500 строк)
  ├── принимает данные от датчиков (CAN/Modbus/MQTT)
  ├── считает HI по той же формуле что на бэкенде
  ├── детектит алерты, генерирует LCAS-события
  ├── пишет в SQLite (локальный буфер до 72ч)
  └── отдаёт по WebSocket на localhost:8000

nginx (localhost)
  └── отдаёт React SPA

sync-agent
  ├── при сети: шлёт накопленные данные на центральный бэкенд (FIFO)
  ├── при отсутствии: копит в SQLite
  └── при восстановлении: синхронизация, диспетчер видит данные задним числом
```

**Дашборд машиниста** подключается к `localhost:8000` — внешняя сеть не нужна.

**Что работает оффлайн (всё критическое):**
- Все параметры в реальном времени (от датчиков напрямую)
- Health Index (расчёт на edge)
- Алерты + LCAS голосовые оповещения (Web Speech API локально)
- Графики трендов (из локального буфера)
- Карта (кэширована, тайлы сохранены)

**Что НЕ работает оффлайн (некритическое):**
- Синхронизация с диспетчером (он не видит этот лок)
- Replay из серверной истории
- Экспорт отчётов на сервер
- Обновление порогов/весов

**Синхронизация при восстановлении:**

```
Сеть вернулась:
  1. Индикатор: "АВТОНОМНЫЙ РЕЖИМ" → "Синхронизация... 847 пакетов"
  2. Отправка буфера (FIFO, хронологический порядок)
  3. Бэкенд пересчитывает, пишет в PostgreSQL
  4. Диспетчер видит "дырку" заполненной
  5. Алерты из оффлайна — видны диспетчеру задним числом
  6. Индикатор: "ONLINE"
```

**UI машиниста в автономном режиме:**

```
┌──────────────────────────────────────────────────────────┐
│ [!] АВТОНОМНЫЙ РЕЖИМ — нет связи с сервером   [* LOCAL] │
│                                                          │
│  Все параметры, HI, алерты — РАБОТАЮТ ЛОКАЛЬНО          │
│  Буфер: 847 пакетов (14 мин 07 сек)                    │
│  Последняя синхронизация: 12:20:33                      │
└──────────────────────────────────────────────────────────┘
```

**Аналогия с авиацией:**

```
Самолёт:   Датчики → Бортовой компьютер → EICAS (локально, ВСЕГДА)
                          ↓
                     ACARS (на землю, может пропасть)

Локомотив: Датчики → Edge Processor → Дашборд (локально, ВСЕГДА)
                          ↓
                     Sync Agent (на бэкенд, может пропасть)
```

**Аргумент для жюри:**

> "Критические данные и алерты никогда не теряются. Даже если машинист
> в тоннеле, в зоне без связи или при отказе сервера — дашборд работает
> автономно. При восстановлении связи данные синхронизируются без потерь.
> Это тот же принцип, по которому работают авиационные системы безопасности."

## Дополнительные концепции из авиации

| Концепция | В авиации | Применение у нас |
|---|---|---|
| **Checklist** | Предполётный чеклист | Чеклист при начале смены — система проверяет все параметры |
